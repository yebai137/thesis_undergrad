## 写作约束

1. 学术表达优先。
   工程实现要转化为“问题定义、方法设计、实验协议、结果分析”的论文语言。
2. 贡献不要发散。
   本科论文创新度以“场景 formalization + targeted improvement + deploy consistency methodology”为主，不夸张包装成通用安全大模型。
3. 章节必须围绕主线服务。
   如果某内容不能支持双任务检测或部署一致性，就不要放进主体章节。
4. 页面增长必须来自有效正文。
   不通过空白附录或无信息重复段落凑页数。
5. 中英文摘要、结论、附录都要可编译。
6. 每次声称“可编译”“已完成”前，都要重新跑编译命令验证。


## 论文视角与二次修改教训

1. 论文必须以作者本人视角写作：作者是“叶伟键，华南理工大学计算机科学与技术全英创新班大四学生”。正文可以使用学术论文常用的“本文/本研究/本课题”，但语义上必须是作者本人完成研究后的陈述，而不是第三方审稿人、指导者、Agent 或答辩顾问在给建议。
2. 严禁出现“若被问及……可以回答……”“这有助于答辩表达”“普通论文/普通应用如何如何，本文可以回答”等面向答辩或修改建议的元叙事。需要表达贡献时，直接写成研究贡献、方法差异或实验结论。
3. 严禁把论文写成修改计划、工程复盘或写作说明。不要在正文中出现“后续补齐”“当前版本”“写法”“证据边界意识”“避免伪装”等过程性语言；未完成内容只能放在“不足与展望”，并以作者对研究局限的学术表述呈现。
4. 从前面修改计划得到的教训：不能为了凑页数加入空泛段落、空表格、待测占位或未完成实验；不能把未实测的跨阶段一致性、阈值扫描、增强消融写成已经完成的结果；不能暴露内部项目名、Phase/test 编号、路径、日期和工程流水线术语。
5. 新主线优先级：YOLO 双目标检测任务建模、海思/昇腾类边缘开发板兼容迁移、部署一致性评价。难例/场景风险只作为应用约束和误差分析维度，不再作为孤立创新点无限扩写。

## AI-research-SKILLs 使用判断

AI-research-SKILLs 只能作为论文结构、证据组织、图表生成和引用核验的辅助工具，不应直接把本科工程论文改写成顶会论文、长期自动研究项目或未完成实验计划。

当前论文不是与 AI-research-SKILLs 完全契合的标准目标场景。该 skills 仓库整体更偏向 NeurIPS/ICML/ICLR、OSDI/SOSP 等顶会论文写作、实验编排和 LLM/多模态/大模型工程；本论文的目标是中文本科毕业论文，主线应保持为：

- 电梯轿厢 `person + ebike` 双目标检测
- YOLO 模型在海思/昇腾类边缘开发板上的兼容迁移
- 部署一致性与可复查实验闭环

### 优先使用的 skills

1. `systems-paper-writing`
   - 用途：借用系统论文的 Problem -> Gap -> Insight -> Design -> Evaluation 结构，强化论文主线与实验论证。
   - 约束：只借结构，不按 OSDI/SOSP 顶会标准包装；不要引入未实现的系统、生产观察、对比实验或性能承诺。
   - 默认提示词方向：
     `请只借用系统论文结构，不按顶会投稿。目标环境 Z=海思/昇腾类边缘开发板，应用 Y=电梯电动车入梯安全视觉，方案 X=YOLO 双目标检测+迁移接口契约+部署一致性验证。基于现有证据重构本科论文主线，不新增未完成实验。`

2. `ml-paper-writing`
   - 用途：审阅摘要、绪论、相关工作、贡献表述和引用纪律。
   - 约束：必须明确“中文本科毕业论文、不可夸大贡献、不可新增实验结果”；引用只能来自 `paper/main.bib`、已有可靠来源，或显式标注为待核验。
   - 默认提示词方向：
     `请按中文本科毕业论文标准审阅摘要、绪论、相关工作和贡献表述。引用只能使用 main.bib 或标注需核验；不要生成虚假 BibTeX；把贡献表述限制为工程方法与评价框架。`

3. `academic-plotting`
   - 用途：基于现有实验表格生成可复现图表建议，例如总体指标柱状图、分类别指标柱状图、阈值敏感性折线图、迁移流程图。
   - 约束：优先使用 `matplotlib`、TikZ 或矢量图；中文标签；适配论文黑白打印；不得捏造缺失数据。
   - 默认提示词方向：
     `请基于 chap05 的表格数据生成可复现图表建议，优先 matplotlib/TikZ/vector，中文标签，适合黑白打印，不捏造缺失数据。`

4. `brainstorming-research-ideas`
   - 用途：只用于一次性收敛创新点，把贡献限定为“场景建模、接口契约、一致性评价”。
   - 约束：不要发散成新课题；不要把难例分类、部署一致性或阈值扫描包装成已完成的全新算法。

5. `ml-training-recipes`
   - 用途：只有在确实要补实验时使用，例如补小样本 `PyTorch/ONNX/OM/board` 对齐、YOLO 变体对比、延迟拆解或训练日志分析。
   - 约束：如果没有真实实验条件或证据文件，不要让它反向生成论文结果。

### 暂不建议使用的 skills

- `0-autoresearch-skill` / `autoresearch`：过重，适合长期自主研究，不适合当前毕业论文收敛阶段。
- `awq`、`gptq`、`hqq`、`bitsandbytes`、`knowledge-distillation`：主要面向 LLM 压缩或通用模型压缩；除非已有真实 YOLO/昇腾量化、剪枝、蒸馏实验，否则不要写入主体贡献。
- `lm-evaluation-harness`、`nemo-evaluator`：面向 LLM 基准评测，不适用于本文目标检测实验。
- `segment-anything`、`clip`、`blip-2`、`llava`：除非用于真实数据标注辅助、可视化分析或后续扩展，否则不要引入主体章节，避免偏离 `person + ebike` 检测主线。

### 使用 skills 前的固定判断

使用任何 AI-research-SKILLs 前，先回答四个问题：

1. 它是在帮助现有论文“收敛主线”，还是在引入新课题？
2. 它需要的实验结果是否已经在 `paper/`、`notes/`、`evidence_index/` 或主工程证据中存在？
3. 它输出的贡献表述是否符合本科毕业论文证据边界？
4. 它是否会诱导生成未核验引用、未完成实验或顶会式夸大表达？

如果答案不明确，默认只用该 skill 做审阅、结构建议或图表建议，不直接改正文结论。
