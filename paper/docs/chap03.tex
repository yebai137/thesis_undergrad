\chapter{YOLO 类双目标检测模型的边缘兼容迁移方法}
\label{cha:method}

本章给出本文的核心方法。与只描述模型训练或只说明板端运行不同，这里把 YOLO 类双目标检测模型从训练权重迁移到目标平台并进入板端运行的过程看作一个需要保持语义一致的系统链路。围绕 HongOU PI（Hi3516DV500）开发板和 YOLOv8n 模型，本文将模型训练、ONNX 导出、OM 编译和板端验证组织为连续链路，使 \textit{person} 与 \textit{ebike} 两类目标的输入、类别、边框、阈值和运行状态都具备可检查的约束。

\section{总体技术路线}

图 \ref{fig:migration_pipeline} 给出了本文方法的总体流程。训练阶段得到的 YOLO 双目标检测权重先经过类别数与检测头适配检查，再导出为 ONNX 中间表示；随后通过 ATC 编译得到边缘设备可执行的 OM 模型；进入板端后，系统完成 NPU 推理、输出解析、阈值筛选和后处理，并与统一评测协议下的结果进行一致性分析。

    \caption{YOLO 双目标检测模型的边缘兼容迁移流程}

该流程的关键不只在模型文件是否能够生成。模型层需要保证检测头输出与双目标任务一致，避免类别数或类别顺序错误；格式层要检查 PyTorch、ONNX 和 OM 之间的输入输出张量语义是否一致；应用层则要让板端后处理逻辑与训练框架中的评价逻辑保持可比较。YOLOv5、YOLOv8 等 YOLO 类检测器在部署时都会涉及输入预处理、类别维度、检测头输出、候选框解码、置信度筛选和 NMS 等环节，因此本文按照这些共性接口组织迁移方法，并在 YOLOv8n 板端链路中展开具体实现与实验分析。只有这些约束同时成立，边缘设备端的检测指标才具有解释意义。

本文将迁移检查组织为语义、数值和运行三个层次。语义检查包括类别数量、类别顺序和输出张量维度；数值检查包括边框坐标、置信度范围和 NMS 后检测数量；运行检查包括模型加载、推理成功率、验证链路端到端耗时和关键分段耗时。这样的顺序先确认基础语义，再讨论 mAP 等检测指标，同时将 batch 外围链路耗时与模型执行阶段耗时分开解释，从而提高实验结论的可信度。

\section{YOLOv8n 场景化训练与验证性能}

为给后续板端结果提供离线参照，本文先使用 Ultralytics YOLOv8n 完成双目标场景化训练。训练在 x86\_64 工作站上进行，工作站配置 Intel Xeon Gold 6226R CPU、约 125 GiB 系统内存和 NVIDIA GeForce RTX 3090 GPU；实际训练采用单张 RTX 3090，未使用多卡并行。数据集标注类别限定为 \textit{person} 与 \textit{ebike}，输入尺寸设为 $640 \times 640$，batch size 为 16，训练轮数为 100 个 epoch，初始权重采用 COCO 预训练 YOLOv8n。训练得到的权重随后依次用于 ONNX 导出、ATC 编译和 OM 模型板端部署，本节验证结果也就成为第五章分析部署后精度回退和能力保留程度的基准。

图 \ref{fig:chap03_training_results} 给出了训练过程中的损失和验证集指标曲线。可以看到，训练后期 box loss、classification loss 和 DFL loss 继续保持下降或趋稳，验证集 precision、recall 和 mAP 指标整体维持在较高水平，未出现明显的训练发散现象。

\section{迁移约束与一致性检查}

训练权重进入开发板之前，需要先把迁移链路中容易被默认处理的假设写清楚。本文把这些假设称为接口契约。它不是额外算法模块，而是对训练、导出、编译和板端运行之间输入、类别、输出、后处理与运行记录的约束说明。对于 YOLO 类检测器，这些约束会落到图像预处理、类别维度、检测头张量、候选框解码、置信度阈值、NMS 策略和板端运行状态上。只有这些环节前后对齐，板端 precision、recall、F1 和 mAP@0.5 才能被解释为同一任务下的检测结果。

输入侧需要核对的是尺寸、颜色通道、归一化、缩放和填充方式。电梯图像常包含强反光和大面积背景结构，若训练阶段和板端的缩放、填充或颜色通道处理不一致，模型对局部电动车结构的响应会发生变化，进而表现为分数漂移或小目标漏检。因此，训练配置、ONNX 输入形状、OM 模型输入和板端预处理路径必须指向同一种图像语义。

类别侧主要核对类别数量、顺序和标签名称。本文采用 \textit{person} 与 \textit{ebike} 两类标签，类别集合较小，但训练配置、导出模型、板端标签表和后处理代码必须使用同一套解释。类别维度或标签顺序一旦错位，框的位置即便画对，报警类别也会被解释错。

输出侧则要弄清检测头张量的维度、坐标格式和置信度含义。训练框架会替使用者完成坐标解码与结果封装，板端应用却必须自己读取张量并判断每一维代表什么。如果中心点、宽高、左上右下角或置信度分数组合方式被误解，最终结果就会表现为边框漂移、类别分数异常或检测数量变化。

后处理契约关注置信度阈值、NMS 阈值和类别处理方式。电梯场景中的反光、门框和局部可见电动车容易生成贴近阈值的候选框，阈值或 NMS 的小幅调整就可能改变最终输出。因此，训练阶段评测脚本和板端应用层需要在阈值含义、NMS 输入、类别过滤和电动车清理策略上保持同一口径。

运行契约约束模型文件、板端二进制、运行模式、成功样本数、失败样本数、端到端处理耗时和分段计时。该契约用于区分模型执行阶段、单帧处理路径和 batch 验证链路，避免把外围链路耗时误解为 OM/NPU 模型执行耗时，也避免在样本未完整处理时报告完整验证结论。

为了让契约落到工程检查中，本文按迁移阶段逐步核对。ONNX 导出后，先看输入尺寸和输出张量形状是否仍符合双目标设置；OM 编译后，再检查板端程序读取到的输出数量和张量维度是否与导出阶段一致；运行验证集时，重点看成功样本数、分类别预测数以及分类别 TP/FP/FN 是否合理。某一项异常时，先回到对应契约定位问题，而不是直接调阈值把现象压下去。

分阶段验证的好处在于每次只处理少数关键问题。训练阶段确认模型本身具备基础检测能力，并保留权重文件与验证集指标；导出阶段核对输入输出形状和类别维度；编译阶段确认 OM 模型可以生成且算子可用；到板端阶段，再检查推理成功率、后处理结果、检测框、分类别指标和运行时信息。这样安排后，板端结果异常时更容易回溯到具体环节。

下面对五类约束的关键一致性环节分别说明。

\subsection{输入预处理对齐}

输入预处理对齐要求训练阶段、导出阶段和板端使用相同的输入尺寸与图像变换语义。对于固定视角电梯场景，目标尺度分布相对稳定，但近景人体和局部电动车会使边缘区域信息非常重要。因此，预处理不应仅追求图像尺寸匹配，还需要保证缩放比例、填充区域和坐标还原方式能够被后处理正确反推。

\subsection{类别语义对齐}

双目标任务的起点是类别语义对齐。本文只包含 \textit{person} 与 \textit{ebike} 两个类别，类别数少并不会降低检查要求，反而要求训练配置、导出模型、板端标签表和后处理代码都采用同一顺序。若这一顺序被读错，检测框位置即便合理，报警对象也会被解释成另一类目标。

分类别 TP、FP、FN 在实验解释中也承担校验作用。类别维度一旦在板端读取错位，\textit{person} 与 \textit{ebike} 的预测数量、召回率和 mAP@0.5 会出现与轿厢经验不符的异常。因此，分类别指标既衡量检测效果，也反过来检验迁移后的类别解释是否成立。

\subsection{输出张量语义对齐}

输出张量语义对齐要求板端程序明确知道 YOLO 检测头输出中每一维的含义。本文关注的双目标检测只有两个类别，但这反而要求类别维度更清晰：如果板端按错误类别数读取张量，类别分数的位置会发生偏移；如果按错误坐标格式解释边框，IoU 指标会下降。本文将输出张量检查作为迁移流程中的必要步骤，以保证后续实验结果能够对应到真实检测行为。

\subsection{后处理一致性}

后处理一致性包括置信度筛选和 NMS 两部分。置信度阈值决定候选框进入最终结果的机会，NMS 决定重叠候选框的保留关系。电梯场景中的反光、门口结构和局部可见目标会产生大量边界候选框，因此阈值稳定性直接影响安全输出。本文在实验章中通过阈值敏感性分析观察不同置信度阈值下 precision、recall、F1 和分类别召回率的变化。

后处理既可以在模型图中完成，也可以在应用层完成。若后处理前移到模型图中，优点是板端应用更简单，缺点是阈值和 NMS 策略调整不够灵活；若后处理保留在应用层，优点是便于针对安全场景调整阈值，缺点是应用层必须正确实现坐标解码和候选框抑制。本文采用应用层后处理思路，并通过契约方式约束其输入、阈值和输出含义。

\subsection{运行时可靠性}

运行时可靠性要求设备端不仅能够加载 OM 模型，还能够完成完整验证集推理并记录成功样本数、失败样本数、验证链路端到端耗时和关键分段耗时。对于安全视觉系统，单张图像推理成功不足以说明系统可用；只有在统一数据集上稳定执行，并能说明耗时指标属于哪一段链路，设备端结果才具备评价意义。

运行时可靠性还需要关注异常样本处理。若某些图像推理失败，实验报告应区分模型检测错误和运行失败；若大量样本触发兜底路径或超时，端到端耗时也需要与成功样本数一起解释。本文在结果章报告成功样本数、失败样本数、batch 图像验证链路平均单图处理耗时和分段计时结果，使精度指标能够与板端运行状态共同解释。

\section{一致性评价方法}

本文用几类证据观察模型迁移后的板端行为。类别语义用于检查不同阶段是否仍输出相同类别含义；边框解释关注坐标还原、IoU 计算和定位漂移；阈值行为观察置信度变化对 precision、recall 和 F1 的影响；运行记录则把样本处理结果和不同耗时口径单独列出。

这些证据并不承担相同角色。类别顺序和分类别 TP/FP/FN 用来判断 \textit{person} 与 \textit{ebike} 的语义是否对齐；IoU、边框漂移和 NMS 后数量用于判断定位变化是否影响检测结果；多阈值 P/R/F1 与 \textit{ebike} recall 反映安全输出是否依赖某个偶然阈值；样本成功/失败记录与分段计时则说明板端验证是否按预期执行，并区分 batch 外围链路与模型执行阶段。

跨阶段逐样本匹配可以作为更细的扩展分析。本文实验章主要报告板端全量验证集结果、分类别结果、阈值后验重算、NMS 实验、后处理消融、视频片段观察和分段计时；若进一步比较 PyTorch、ONNX、OM 与板端四阶段输出，可以逐样本分析类别、坐标和分数在迁移链路中的变化。这样区分后，正文结论与已有证据范围保持一致。

样本级匹配的思路并不复杂：对同一张输入图像，先按类别过滤候选框，再按 IoU 从高到低匹配同一目标，并比较不同阶段的类别、坐标和分数。若类别一致但 IoU 明显下降，问题多半出在坐标还原或 NMS；若 IoU 接近但类别不同，更应检查类别顺序或输出维度；若检测数量突然变化，则回到阈值和候选框抑制逻辑。这样可以把跨阶段问题转化为具体检查项。

结果异常时，排查顺序也按契约展开。类别大量错位时，先检查类别顺序和输出张量维度；边框整体偏移时，先检查缩放、填充和坐标解释；检测数量突变时，回到 conf、NMS 和类别清理策略；时延异常升高时，再看板端执行路径、输入输出复制和资源状态。这个顺序的作用，是让部署问题能够定位到具体技术环节。

\section{本章小结}

本章给出了 YOLO 类双目标检测模型进入目标平台前后的检查流程。论文把输入预处理、类别配置、输出张量、后处理参数和运行状态放入同一迁移链路中，目的不是增加新的算法模块，而是保证板端结果能够回到具体环节进行核对。与只报告模型精度相比，这种写法更关注模型进入开发板后的输出含义和实验记录。
