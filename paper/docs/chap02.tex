\chapter{技术基础与相关工作}
\label{cha:background}

本章围绕后续方法章所需的基础知识展开。由于本文的核心问题不是单纯训练一个检测模型，而是使 YOLO 双目标检测模型在海思/昇腾类边缘开发板上保持可解释、可复查的部署行为，因此本章从目标检测、YOLOv8、边缘 AI 部署和部署一致性四个方面建立技术背景。

\section{目标检测基础}

目标检测需要同时回答“目标在哪里”和“目标属于哪一类”两个问题。与图像分类只输出整幅图像类别不同，目标检测需要输出一组边界框、类别标签和置信度分数。对于电梯轿厢场景，本文只关注 \textit{person} 与 \textit{ebike} 两类目标，但检测任务并未因此变得简单。狭小空间中的遮挡、反光和近距离透视会使同一目标在不同帧中呈现不同尺度和形态，进而影响边框回归和类别置信度。

评价目标检测模型时，常用指标包括 precision、recall、F1、IoU 和 mAP@0.5。precision 表示预测为目标的结果中有多少是真实目标，recall 表示真实目标中有多少被成功检测，F1 则综合二者。当预测框与真实框的交并比达到阈值且类别一致时，该预测通常被视为正确检测。本文采用 mAP@0.5 评价图像级检测能力，并在安全视觉分析中同时关注分类别召回率，因为电动车漏检比普通背景误检更直接影响预警效果。

mAP@0.5 适合概括一批图像上的平均检测能力，但它不能单独回答所有部署问题。首先，mAP 是离线标注集上的统计量，无法说明板端程序是否正确解释了类别顺序和输出张量。其次，平均指标可能掩盖分类别差异，例如 \textit{ebike} 与 \textit{person} 在遮挡、尺度和安全含义上并不相同。最后，mAP 本身不包含运行成功率、耗时口径和后处理阈值变化等信息。因此，本文在第五章同时报告总体指标、分类别指标、阈值后验重算和分段计时，使检测性能与部署行为能够相互印证。

设某一类别的真正例、假正例和假负例分别为 TP、FP 和 FN，则 precision、recall 与 F1 可以写为
\begin{equation}
    P=\frac{TP}{TP+FP},\quad R=\frac{TP}{TP+FN},\quad F1=\frac{2PR}{P+R}.
\end{equation}
上述三个指标在安全视觉中的含义并不完全相同。precision 偏低意味着误报较多，可能造成无效提醒；recall 偏低意味着漏检较多，可能造成风险未被发现；F1 则适合观察二者的折中关系。对于电动车入梯场景，本文更关注 \textit{ebike} 类别的召回率是否在阈值变化下保持稳定。

目标检测结果还受到阈值和非极大值抑制的影响。置信度阈值决定低分候选框是否保留，NMS 阈值决定重叠候选框是否被抑制。若阈值设置过高，系统可能漏掉局部可见的电动车；若阈值设置过低，又可能引入门框、反光或大面积背景误检。因此，阈值并不是简单的工程参数，而是连接检测性能和安全输出稳定性的重要因素。

电梯轿厢目标检测与开放场景检测存在明显差异。开放场景通常目标种类多、背景变化大，评价重点更多放在平均精度；电梯场景类别较少，但背景结构固定且反复出现，误报和漏报往往集中在少数高风险区域。表 \ref{tab:elevator_detection_diff} 概括了二者的差异。本文采用双目标检测而不是通用多类别检测，正是为了让模型和评价协议聚焦于电动车入梯风险。

\section{YOLO 系列检测器与 YOLOv8 特点}

YOLO 系列检测器将目标检测建模为单阶段预测问题，直接从输入图像生成边框、类别和置信度结果，具有结构紧凑、推理流程清晰和部署友好等特点\cite{redmon2017yolo9000}。与两阶段检测器相比，YOLO 系列更适合实时或准实时场景，这也是其在边缘视觉任务中被广泛采用的重要原因。

从结构上看，YOLO 检测器通常包含 Backbone、Neck 和 Head 三部分。Backbone 负责提取视觉特征，Neck 负责融合不同尺度的特征，Head 负责输出检测框与类别分数。电梯场景中既存在近景人体，也存在局部可见的电动车结构，因此多尺度特征融合对稳定检测具有重要意义。图 \ref{fig:yolo_pipeline} 给出了本文采用的 YOLO 双目标检测流程。

\begin{figure}[htbp]

    \caption{YOLO 双目标检测流程}

YOLOv8 进一步采用较轻量的模型尺度和更便于部署的检测头设计。对于本文的双目标任务，模型只需要输出 \textit{person} 与 \textit{ebike} 两类分数，但导出后的张量维度、坐标顺序和类别顺序必须与板端程序保持一致。若板端仍按通用类别数或旧输出格式解释结果，即使模型权重本身正确，最终检测框也可能出现类别错位或坐标失真。

近年来，DETR 等端到端检测框架将目标检测建模为集合预测问题，减少了手工设计候选框和部分后处理步骤\cite{carion2020detr}。这类方法展示了目标检测结构的另一条发展路线，但在本文的边缘开发板场景中，模型部署成熟度、算子支持和实时处理成本同样重要。相比之下，YOLO 系列在工程链路中具有更成熟的导出、编译和板端适配经验，更适合作为本科论文中可复查的边缘安全视觉实现基础。

从部署角度看，YOLOv8 的优势在于主干、特征融合和检测头之间边界清晰，便于检查导出模型的输入输出。但这种清晰结构并不意味着部署可以自动正确完成。训练框架中的预测结果通常已经完成坐标还原、类别筛选和 NMS，而导出模型可能只给出未经封装的张量。板端程序必须明确张量维度、类别数、分数含义和坐标尺度，才能得到与训练端可比较的检测框。

轻量化模型选择需要兼顾精度和设备约束。MobileNet、EfficientNet、EfficientDet 等研究说明了特征提取效率、模型缩放和检测性能之间的关系\cite{howard2019mobilenetv3,tan2019efficientnet,tan2020efficientdet}。在电梯场景中，类别数少并不意味着可以忽略模型表达能力；遮挡和局部可见目标要求模型保留足够多尺度特征。同时，边缘开发板的算力和算子支持又要求模型结构不过度复杂。因此，本文采用轻量 YOLO 模型作为检测基础，并把可部署性纳入模型选择依据。

在本文任务中，YOLO 失效通常不是随机出现的，而是与轿厢空间结构有关。门口区域的垂直边缘可能被误识别为目标边界，镜面反光可能生成重复候选框，电动车只露出车把或车轮时可能造成低置信度候选。由此可见，模型结构、数据分布和后处理阈值共同决定最终效果。后续实验中对分类别召回率和阈值敏感性的关注，正是为了把这些失效现象转化为可分析指标。

\section{海思/昇腾类边缘 AI 部署基础}

边缘智能强调在靠近数据源的位置完成推理，以降低带宽、隐私和响应时延压力\cite{shi2016edge}。海思/昇腾类边缘开发板通常通过专用 NPU 执行神经网络推理，模型需要先从训练框架导出为中间表示，再通过 CANN/ATC 等工具编译为设备可执行的 OM 离线模型。图 \ref{fig:ascend_deploy_pipeline} 展示了本文关注的部署链路。

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth]{image/fig3-ascend-deploy.png}
    \caption{海思/昇腾类开发板上的 YOLO 部署链路}
    \label{fig:ascend_deploy_pipeline}
\end{figure}

该链路通常包括五个环节。第一，训练端得到 YOLO 权重；第二，导出 ONNX 中间表示；第三，使用 ATC 编译得到 OM 模型；第四，在板端通过 NPU 运行推理；第五，在应用层完成输出解码、阈值筛选和 NMS 后处理。每个环节都可能改变输出行为。例如，预处理方式不同会改变输入分布，张量读取顺序错误会改变类别语义，后处理阈值不同会改变最终检测数量。

在海思/昇腾类平台上，AIPP、输入尺寸、算子支持和内存布局都会影响模型执行。AIPP 若承担颜色空间转换或归一化操作，则训练端预处理必须与板端配置对应；若预处理在应用层完成，则应用层需要保证输入张量与模型期望一致。OM 模型编译完成后，其输入输出名称、形状和数据类型也需要被记录，否则后续调试很难判断错误来自模型本身还是应用层解释。

模型压缩和量化是边缘部署中的常见方法。整数推理友好量化、剪枝和知识蒸馏分别从数值表示、结构冗余和知识迁移角度降低模型部署成本\cite{jacob2018quantization,han2016deep,hinton2015distilling}。混合精度量化研究进一步表明，不同网络层对量化误差的敏感性并不相同，量化策略会影响最终精度和部署效率\cite{dong2019hawq}。不过，本文并不把模型压缩作为主要创新，而是关注自训练 YOLO 模型在既定边缘平台上的兼容迁移与行为一致性。

边缘部署还需要区分“能够运行”和“适合应用”。能够运行表示模型可以被编译、加载并完成推理；适合应用则要求输出稳定、运行口径清晰、失败样本可解释，并且后处理规则能够服务安全需求。本文实验结果中的成功样本数、batch 验证链路耗时、分段计时和阈值敏感性分析，就是为了避免只用“模型已部署”来替代系统层面的可用性判断。

\section{相邻视觉任务与本文边界}

电梯安全视觉可以延伸出多种任务形态，包括人数统计、目标跟踪、姿态估计和视频行为理解等。人数统计研究能够反映轿厢拥挤程度，并已有基于边缘轻量检测器的电梯人数统计系统\cite{lin2023elevatorcounting}；多目标跟踪方法如 DeepSORT 和 ByteTrack 则关注跨帧关联与轨迹保持\cite{wojke2017deepsort,zhang2022bytetrack}；姿态估计和视频识别方法可以进一步描述人体关键点或动作模式\cite{cao2017openpose,feichtenhofer2019slowfast}。这些方向都与电梯安全有关，但它们需要额外标注、时序建模或行为定义。

本文没有把上述任务全部纳入主体实验，而是将研究范围限定在 \textit{person} 与 \textit{ebike} 双目标检测及其边缘迁移上。这样处理有两个原因。第一，电动车入梯治理首先需要稳定回答“人和电动车是否被检测到”，这是后续计数、跟踪和行为判断的基础。第二，本科论文需要保证实验结论可复查，如果同时扩展到跟踪或动作识别，就需要额外的视频级人工标注和时序评价协议。本文因此将代表性视频片段作为运行状态和输出稳定性的补充观察，而不把它扩展为完整视频理解任务。

\section{部署一致性问题}

部署一致性是指同一检测模型经过格式转换和设备迁移后，其输入语义、输出语义和后处理行为仍保持可比较状态。对于普通离线实验，研究者往往只关心最终 mAP；但在安全视觉系统中，类别错位、边框漂移、阈值变化和运行时异常都可能导致预警结果变化。因此，部署一致性应被视为方法的一部分，而不只是实验环境说明。

本文将一致性问题分为四类。第一是类别一致性，即训练端、ONNX 端、OM 端和板端对 \textit{person} 与 \textit{ebike} 的类别顺序解释一致。第二是边框解释一致性，即坐标归一化、缩放和裁剪方式在评价设计上保持可比较。第三是阈值稳定性，即置信度阈值和 NMS 参数在不同阶段含义一致。第四是运行时可靠性，即模型能够在边缘设备上稳定执行，并记录成功样本数、失败样本数、batch 端到端耗时和分段计时等运行状态。

这四类一致性之间存在依赖关系。类别一致性是结果解释的前提，若类别顺序错误，后续 precision 和 recall 都失去意义；边框解释一致性是 mAP 和 IoU 评价的前提，若坐标还原错误，模型可能被误判为定位能力不足；阈值稳定性决定安全输出是否依赖偶然参数；运行时可靠性则决定一次实验是否能够代表真实设备行为。因此，本文将一致性评价放在方法章和实验章之间，作为连接模型迁移与实验结论的关键桥梁。

已有部署研究已经指出，深度模型从训练框架迁移到生产环境时，格式转换、算子支持和运行时差异会引入额外风险\cite{stacker2021deployment,chen2021aqd}。在电梯安全视觉中，这类风险会进一步受到场景因素放大。例如，反光和门口结构可能使候选框分数接近阈值，局部可见电动车可能使边框定位更敏感，拥挤重叠则可能改变 NMS 结果。因此，本文后续方法章将围绕接口契约和分阶段验证建立部署一致性分析框架。

\section{相关工作小结}

综上，YOLO 系列检测器为电梯双目标检测提供了高效基础，边缘 AI 平台为本地安全预警提供了硬件条件，模型压缩与部署研究则说明了端侧推理的可行性。然而，现有工作往往分别讨论检测精度或边缘部署，较少把训练权重、ONNX 中间表示、OM 离线模型和板端后处理放入同一评价框架。本文围绕这一缺口展开：在电梯轿厢场景中，以行人与电动车双目标检测为任务载体，建立面向海思/昇腾类边缘开发板的兼容迁移与一致性评价方法。
