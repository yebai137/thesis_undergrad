\chapter{实验结果与分析}
\label{cha:experiments}

本章围绕第四章定义的实验协议，对 YOLO 双目标检测模型在边缘设备端的表现进行分析。实验重点不是单纯报告一个离线精度数值，而是先通过官方基线、YOLOv8n 场景化训练结果和板端 FP16/OM 结果建立参照系，再围绕七个评价维度展开：第一，模型迁移到板端后是否能在完整验证集上保持稳定检测能力；第二，分类别输出是否符合 \textit{person + ebike} 双目标任务语义；第三，不同置信度阈值下安全输出是否稳定；第四，NMS 参数变化是否会改变重叠候选框的保留关系；第五，应用层后处理策略是否影响两类目标的误报与召回；第六，代表性视频片段中连续输出是否稳定；第七，batch 图像验证链路耗时、单帧核心处理耗时和模型执行阶段耗时是否能够被清楚区分。

\section{实验设置与评价范围}

本文实验使用同一轻量 YOLO 双目标检测模型，类别为 \textit{person} 与 \textit{ebike}。图像验证集包含 720 张电梯轿厢图像，评价指标包括 precision、recall、F1、mAP@0.5、分类别指标和板端运行状态。板端推理使用搭载华为海思 Hi3516DV500 视觉 SoC 的 HongOU PI 开发板完成，模型以 OM 离线模型形式运行，后处理包含置信度筛选和 NMS。

实验前先确认板端验证环境、模型文件、应用程序和数据集均处于就绪状态。完整验证集已完成板端实测，阈值敏感性分析采用既有板端检测输出进行后验筛选重算；NMS 和后处理消融采用同一板端 campaign 协议，保证只有一个参数变化；分段计时实验则使用新增计时埋点对小样本 batch 图像验证和代表性视频片段进行补充复查。

阈值后验重算的依据是板端检测输出中保留了每个候选框的类别、置信度和坐标，同时每张图像具有对应真实标注。因此，在原始候选阈值低于待分析阈值的前提下，可以通过重新筛选候选框并按 IoU@0.5 匹配真实框，得到不同置信度阈值下的 precision、recall 和 F1。该做法不改变模型推理结果，只改变结果解释阈值，适合用于分析安全输出对阈值选择的敏感程度。

本章评价问题与证据形式保持一一对应。RQ1 使用 720 张图像完整验证总体指标回答板端完整验证是否可执行且具备检测能力；RQ2 使用分类别 TP、FP、FN、recall 和 mAP@0.5 检查两类目标是否保持正确语义；RQ3 使用 0.20--0.40 置信度阈值后验重算分析安全输出稳定性；RQ4 和 RQ5 分别使用固定 score 后改变 NMS 阈值、改变电动车清理策略的板端 campaign；RQ6 使用代表性视频片段和平滑窗口对比观察连续输出；RQ7 使用 50 张图像 batch 复查与连续视频分段计时解释运行时指标。这样的组织方式用于保证每个实验结论都能追溯到具体数据类型，避免把不同口径的指标混为一谈。

\section{训练阶段与板端同集对比分析}

为了避免将板端指标孤立理解，本文进一步把第五章结果放入同一任务证据链中解释。该证据链包含三类参照：第一，官方 YOLOv8n COCO-person 基线，用于说明通用预训练模型直接迁移到电梯俯视场景时的不足；第二，基于 RTX 3090 GPU 完成训练的 YOLOv8n 场景化模型，用作训练阶段能力参照；第三，Hi3516DV500 板端 FP16/OM 完整验证结果，用于衡量部署后检测能力是否被保留。三者共同回答的问题不是“本文指标在所有公开数据集上排名如何”，而是在同一电梯验证集和同一安全任务背景下，模型从通用基线、场景训练到板端部署的效果变化是否具有解释性。

\section{RQ3：阈值变化是否影响安全输出}

为分析置信度阈值对安全输出的影响，本文在既有板端检测输出基础上进行后验筛选重算。由于原始板端运行使用的候选阈值低于本节扫描阈值，本文可以在不重新推理的条件下，分别保留置信度不低于 0.20、0.25、0.30、0.35 和 0.40 的检测框，并按 IoU@0.5 重新统计 precision、recall 和 F1。表 \ref{tab:threshold_sensitivity_results} 给出了结果。

结果显示，随着置信度阈值从 0.20 提高到 0.40，总体 precision 从 0.911 提升到 0.942，recall 从 0.919 下降到 0.903，F1 保持在 0.915--0.922 区间。电动车召回率在五个阈值下保持在 0.953 以上，说明在当前验证集和板端输出条件下，电动车检测对这一阈值区间相对稳定。行人召回率随阈值提高从 0.884 降至 0.857，说明行人类别中存在更多接近阈值边界的候选框。

\section{RQ4：NMS 参数如何影响候选框保留}

NMS 参数敏感性实验用于观察重叠候选框的保留关系是否依赖单一 IoU 阈值。本文设计的实验固定模型、验证集、置信度阈值和后处理策略，只改变 NMS 阈值，取值为 0.35、0.45 和 0.55。低 NMS 阈值会更强地抑制重叠框，可能减少重复框但增加遮挡区域漏检；高 NMS 阈值会保留更多相邻框，可能提高局部目标召回但增加误报和重复框。

该实验与阈值后验重算不同，需要重新运行板端推理，因为 NMS 在板端应用层已经影响最终输出。三组实验均使用同一 OM 模型、同一板端二进制和同一验证集切分，且均完成 720 张图像处理，失败样本数为 0。完整 campaign 标识和路径见实验产物索引；其中 NMS=0.35 的前序尝试曾出现板端 VDEC 启动异常导致样本不完整，因此正文只采用补跑后样本完整的 \texttt{r4} 结果。

结果显示，NMS 从 0.35 提高到 0.55 时，预测框总数从 1526 增加到 1542，FP 从 146 增加到 160，FN 从 116 降至 114。行人类别 F1 在三组设置下均为 0.878，说明行人结果在该区间内几乎不受 NMS 参数影响；电动车 F1 则从 0.952 下降到 0.944，同时电动车召回率从 0.960 小幅上升到 0.963。该趋势表明，提高 NMS 阈值会保留更多电动车候选框，略微减少漏检，但也引入更多误报，使 F1 略有下降。

\begin{figure}[htbp]

    \caption{NMS 参数敏感性结果}

从安全场景角度看，NMS=0.45 是较平衡的设置：其 mAP@0.5 为 0.910，整体 FP/FN 分别为 152 和 115，与 0.55 的 mAP@0.5 基本相当，但误报更少；与 0.35 相比，0.45 保留了略多候选框并取得更高 mAP@0.5。该结果说明，本文使用的 NMS 设置不是孤立参数，而是在召回、误报和重复框保留之间取得折中。

\section{RQ5：后处理策略消融如何界定部署模块作用}

应用层后处理消融用于回答一个容易混淆的问题：板端输出变化究竟来自模型权重，还是来自模型之后的候选框清理逻辑。本文将电动车误检清理策略显式暴露为 \texttt{full}、\texttt{safe} 和 \texttt{off} 三种模式。其中，\texttt{full} 表示完整启用误检清理，\texttt{safe} 表示只启用较保守的重复框和明显条带清理，\texttt{off} 表示关闭电动车类别特定清理。默认 \texttt{auto} 模式保持当前板端行为，即 batch 评价路径保留更原始的 \texttt{safe} 输出，视频 file 路径使用更面向展示的 \texttt{full} 输出。

该消融不是训练消融，也不表示本文提出了新的检测网络结构。它的作用是把板端应用层规则从“隐含工程细节”转化为可以复查的部署模块。三组实验均完成 720 张验证图像，失败样本数为 0，产物分别对应 \texttt{cleanup\_full}、\texttt{cleanup\_safe} 和 \texttt{cleanup\_off} campaign。

结果表明，\texttt{full} 策略将电动车 FP 从 \texttt{safe} 的 47 降至 38，但电动车 FN 从 28 增至 169，电动车 recall 从 0.961 降至 0.765，F1 从 0.949 降至 0.842。这说明完整清理策略确实减少了部分电动车误报，但代价是过度抑制真实电动车候选框。相比之下，\texttt{safe} 策略保留较高电动车召回率；\texttt{off} 策略进一步将电动车 recall 提高到 0.963，但 FP 增至 55，F1 降至 0.944。

上述结果说明，安全视觉系统的最终输出不仅取决于模型权重，还受到应用层后处理策略影响。对于本文的图像验证链路，\texttt{safe} 策略在误报控制和召回保持之间更平衡，因此更适合作为板端 batch 评价的默认口径；\texttt{full} 策略可用于展示链路中的强清理，但不宜直接作为图像级检测指标的默认设置。正文评价以完整验证集统计为主，定性样例和视频片段用于补充说明输出形态与连续性。

\section{RQ6：代表性视频片段如何补充稳定性观察}

除图像验证集外，本文还使用代表性视频片段观察连续场景下的输出稳定性。机器侧统计显示，在一个约 14.58 s、437 帧的代表性片段中，系统能够记录原始检测输出、筛选后输出、分段计时和运行资源信息。视频片段评价采用检测保持率、闪烁窗口、连续无输出片段和分段计时等机器侧指标，用于观察连续帧中的检测保持、闪烁、漏检现象和运行时状态。

视频稳定性实验固定模型、score 和 NMS，只改变平滑窗口，重点比较窗口 1 与窗口 5。前者接近无平滑或最小平滑，能够暴露单帧检测分数波动；后者代表默认连续输出策略，更接近真实展示链路。两组实验均处理 437 帧，产物分别为 \texttt{video\_smooth1} 和 \texttt{video\_smooth5}。

从逐帧输出看，两组实验的检测保持率均为 0.982，均出现 3 个 \textit{person\_flash} 闪烁窗口，闪烁帧数均为 5，连续无输出片段数均为 2，最长无输出片段为 5 帧。窗口 1 与窗口 5 在该代表性片段上的稳定性统计基本一致，说明当前片段中主要波动不是简单通过增大平滑窗口即可消除的单帧噪声。两组视频路径的 \texttt{frame\_proc\_ms} 均值分别为 11.811 ms 和 11.951 ms，\texttt{model\_execute\_ms} 均值分别为 8.359 ms 和 8.365 ms，均低于 30 fps 对应的 33.3 ms 帧间隔。

\section{部署一致性综合分析}

从训练阶段和板端对比看，官方 COCO-person 基线不足以直接支撑电梯俯视场景检测，YOLOv8n 场景化训练显著提高了同集检测能力；板端 FP16/OM 结果相比训练阶段验证结果存在约 0.078 的 mAP@0.5 回退，但仍保持 0.910 的 mAP@0.5。该结果说明，本文的板端指标既不是孤立数值，也不是训练阶段指标的简单复制，而是部署链路实际执行后的实测结果。

从完整验证集结果看，板端模型能够成功处理全部 720 张图像，并给出与双目标任务一致的分类别输出。该结果支持输入契约、类别契约、输出契约和运行契约在当前板端验证范围内未出现明显失配。特别是 \textit{person} 与 \textit{ebike} 两类均能形成可解释的 TP、FP、FN 统计，表明板端后处理没有出现类别顺序错位或输出张量无法解释的问题。

从阈值敏感性结果看，电动车召回率在 0.20--0.35 阈值区间内变化较小，说明电动车检测结果在当前验证集上具有一定阈值稳定性。相比之下，行人召回率下降更明显，说明近景遮挡和多人重叠场景仍可能产生低置信度候选框。该差异与第四章的场景复杂性分析一致，也说明场景复杂性因素可以用于解释部署后的指标变化。

从 NMS 敏感性结果看，0.35、0.45 和 0.55 三组设置均能完成 720 张图像验证，且总体指标变化较小。NMS 阈值提高后，电动车候选框数量和 FP 略有增加，FN 略有下降，说明 NMS 参数主要影响电动车相邻候选框的保留关系。0.45 在 mAP@0.5、FP 和 FN 之间表现更均衡，适合作为本文主要实验设置。

从后处理消融结果看，完整清理策略虽然减少电动车 FP，但显著增加电动车 FN，导致 recall 降至 0.765。相反，\texttt{safe} 与 \texttt{off} 都保持较高电动车召回率，其中 \texttt{safe} 相比 \texttt{off} 少产生 8 个电动车 FP。该结果说明后处理策略属于部署链路中的重要变量，必须与模型权重结果分开解释。

从视频稳定性结果看，代表性视频片段中窗口 1 和窗口 5 的检测保持率、闪烁窗口数和最长无输出段一致，说明该片段中的稳定性问题不能简单归因于平滑窗口设置。该实验补充连续输出观察，与图像验证集检测指标共同解释部署后的稳定性。

从分段计时结果看，batch 图像验证链路端到端耗时与模型执行阶段耗时必须分开解释。完整验证集中的 1119.510 ms 表明 batch 验证链路可以完整复查，但不代表 OM 模型执行阶段本身需要约 1.1 s。新增分段计时补上了这一解释缺口，使本文能够同时报告可复查的 batch 验证结果和更细粒度的板端运行时拆解。

本文关于部署一致性的结论建立在完整板端验证、分类别统计、阈值后验重算、NMS 敏感性、后处理消融、视频稳定性观察和分段计时之上。PyTorch、ONNX、OM 和板端四阶段的逐样本统一匹配提供进一步分析边框漂移的细粒度路径。

\section{场景复杂性与误差解释}

结合第四章的场景复杂性因素，可以进一步解释实验结果中的分类别差异。行人类别的误差主要来自近景截断、多人重叠和门口区域干扰。当人体靠近摄像头或只出现局部身体结构时，边框可能与真实框重合不足；当多人重叠时，NMS 可能抑制部分候选框，造成漏检或定位偏差。电动车类别虽然整体召回率较高，但局部可见样本仍然是主要困难来源，尤其是只出现车把、车轮或车身边缘时，置信度更容易接近阈值边界。

阈值敏感性结果也支持这一解释。电动车召回率在 0.20--0.35 阈值区间内变化较小，说明大部分电动车候选框分数较稳定；行人召回率随阈值升高持续下降，说明行人类别存在更多低置信度但可能正确的候选框。由此可见，阈值选择不能只依据总体 F1，还需要考虑不同类别在安全任务中的风险权重。

从部署一致性角度看，场景复杂性分析的价值在于帮助判断误差是否来自模型能力不足，还是来自迁移和后处理过程。如果某类复杂样本在所有阶段都表现困难，问题更可能来自数据和模型表达；如果同一复杂样本在训练阶段结果正确而板端异常，则需要优先检查输入契约、输出契约或后处理契约。本文通过板端完整验证、阈值重算、NMS 敏感性、后处理消融、视频稳定性观察和分段计时建立了后续跨阶段分析的基础。

\section{结果讨论与扩展方向}

综合上述结果，本文方法的有效性体现在三个方面。第一，官方基线、YOLOv8n 场景化训练结果和板端 FP16/OM 结果共同说明，场景化训练能够显著提升电梯双目标检测能力，且板端部署后仍保留主要检测效果。第二，板端完整验证集结果表明 YOLO 双目标检测模型能够完成从训练权重到边缘设备执行的兼容迁移。第三，分类别、阈值、NMS、后处理消融、视频稳定性观察和分段计时结果说明，本文不仅关注检测指标，也关注安全阈值、应用层规则、连续输出状态和运行时口径是否可解释。

围绕进一步应用，后续可从三个方向推进。第一，batch 图像验证链路端到端耗时较高，可进一步优化图像读取、数据搬运和外围调度，使实验验证链路与连续视频链路在运行口径上更加接近。第二，代表性视频片段可扩展为带逐帧标注的视频集，从而把连续输出稳定性与视频级检测精度结合起来。第三，跨阶段逐样本一致性可进一步量化每个目标从训练阶段到板端部署阶段的边框漂移，使兼容迁移评价更加细粒度。

\section{本章小结}

本章报告了 YOLO 双目标检测模型的训练阶段与板端同集对比、边缘设备端完整验证结果、分类别结果、阈值敏感性后验重算、NMS 敏感性、后处理消融、视频稳定性和分段计时结果。实验表明，YOLOv8n 场景化训练模型达到 0.988 的 mAP@0.5，板端 FP16/OM 在 720 张验证图像上取得 0.910 的 mAP@0.5，其中行人类别为 0.869、电动车类别为 0.952，两类目标在多个置信度阈值和 NMS 设置下均保持较稳定的召回率。后处理消融说明完整清理策略会显著牺牲电动车召回率，\texttt{safe} 策略更适合作为图像验证链路默认口径；视频实验补充说明连续输出状态和分段耗时。上述结果支撑了本文关于兼容迁移和部署一致性评价的主要结论。
