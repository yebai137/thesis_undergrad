\chapter{结论与展望}
\label{cha:conclusion}

本文面向电梯轿厢场景中的电动车入梯风险，研究了基于 YOLO 的行人与电动车双目标检测模型及其在 HongOU PI（Hi3516DV500）开发板上的兼容迁移问题。与只关注离线检测精度的研究不同，本文将训练权重、ONNX 中间表示、OM 离线模型、板端推理和后处理输出纳入同一评价框架，重点分析模型迁移后类别语义、边框解释、阈值行为和运行状态是否具备可复查的一致性。

本文的主要工作可以概括为三点。第一，面向电梯轿厢建立了 \textit{person + ebike} 双目标检测任务，并结合门口结构、局部可见、拥挤重叠和光照变化等场景因素解释检测误差来源。第二，完成了 YOLOv8n 场景化自训练，并将 Linux FP32 训练端结果作为板端部署结果的参照上限。第三，提出了自训练 YOLO 模型到 HongOU PI（Hi3516DV500）开发板的兼容迁移方法，围绕输入预处理、类别顺序、输出张量、后处理参数和运行状态建立接口契约，并通过完整验证集结果、分类别指标、阈值敏感性后验重算、NMS 敏感性、后处理消融、分段计时和代表性视频稳定性统计分析迁移后的实际行为。

实验结果表明，官方 YOLOv8n COCO-person 基线在修正评估口径下的 F1 为 0.736，而场景化自训练后的 Linux FP32 模型在同一验证集上的 mAP@0.5 为 0.988，说明电梯场景需要专门训练支撑。在 720 张图像验证集上，边缘设备端总体 precision 为 0.901，recall 为 0.923，F1 为 0.912，mAP@0.5 为 0.910，板端 batch 图像验证链路端到端平均单图处理耗时约为 1119.510 ms。其中，行人类别 mAP@0.5 为 0.869，电动车类别 mAP@0.5 为 0.952。板端 mAP@0.5 相比 Linux FP32 训练端下降约 0.078，但仍保持较高检测能力，说明部署链路带来可量化回退的同时保留了主要识别能力。阈值敏感性后验重算进一步显示，在 0.20--0.40 置信度阈值范围内，总体 F1 保持在 0.915--0.922 区间，电动车召回率保持在 0.953 以上。NMS 敏感性实验显示，0.35、0.45 和 0.55 三组设置均完成 720 张图像验证，NMS=0.45 在 FP、FN 和 mAP@0.5 之间较为均衡。后处理消融显示，完整电动车清理策略会将电动车 recall 降至 0.765，而 \texttt{safe} 策略能保持 0.961 的电动车 recall，更适合作为图像验证链路默认口径。代表性视频片段实验显示，窗口 1 与窗口 5 的检测保持率均为 0.982，均出现 3 个闪烁窗口，说明该片段中的连续输出波动不能简单依靠增大平滑窗口消除。分段计时结果表明，50 张图像 batch 复查中的模型执行阶段均值约为 8.208 ms，代表性视频片段中的模型执行阶段均值约为 8.407 ms，说明 batch 端到端验证耗时不能等同于 OM/NPU 模型执行阶段耗时。上述结果说明，本文方法在当前验证集与代表性片段范围内表现出较稳定的双目标检测能力，并形成可复查的部署一致性评价证据。

本文的应用价值在于为电梯边缘安全视觉提供了一条可复查的落地路径。电梯轿厢场景的目标类别并不复杂，但系统必须在边缘设备上稳定运行，并且能够说明输出结果来自哪一组模型、阈值和后处理规则。本文将 YOLO 检测模型与 HongOU PI（Hi3516DV500）开发板部署链路连接起来，使检测精度、阈值行为和运行状态能够在同一协议下被分析。这种思路也可为门厅、楼道、地下车库等封闭式公共空间的边缘视觉任务提供参考。

本文的研究边界主要包括四点。首先，当前实证结果主要集中在板端完整验证、参数敏感性、后处理消融、视频稳定性观察和分段计时，训练端、ONNX 端、OM 端和板端之间的逐样本统一匹配可作为后续扩展。其次，batch 图像验证链路端到端耗时较高，当前分段计时主要用于解释运行时口径，而不是替代完整验证集结果。再次，代表性视频片段用于连续输出稳定性观察，视频级 precision、recall 或 mAP 需要额外逐帧人工标注支持。最后，本文聚焦于行人与电动车双目标检测，人数统计、跌倒检测或其他异常行为识别可在同一接口契约和一致性评价框架下继续拓展。
